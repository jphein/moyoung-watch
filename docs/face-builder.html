Concept

How a custom face works

A face is a binary container: a header, a list of fields (each a type code + on-screen rectangle), and the bitmap blobs the fields draw. You supply the artwork; the watch supplies the data.

Design the fields.

Pick field types (e.g. BACKGROUND, TIME_H1..M2, STEPS, HR, BATT) and lay out each one's rectangle. Digit fields need a strip of glyph bitmaps (usually 10, for 0–9); icon/progress fields need their frames.

Pack the .bin.

Bitmaps + layout compile into the MoYoung binary (Type A/B/C — B/C use a full-screen background and per-field placement). david47k/dawft builds and dumps these; our CLI's inspect reads the header back.

Flash over Bluetooth.

Streamed to the watch over GATT service 0xFEEA — announce size, push the image in MTU-sized chunks, wait for the completion ack, activate the slot. No app, no cloud.

The watch fills in the values.

Once installed, the firmware paints the current time / steps / HR / battery / weather into the fields you placed. You're designing where and how each value appears — not sending the values.

Binary layout

Inside a face .bin

The header is fixed; each field is a type code plus a placement rectangle; the blobs (bitmaps) follow. Type B/C use a full-screen background and 16-bit coordinates; the older Type A tiles the background in strips.

# header
byte 0      fileID       0x04 = Type A · 0x81 / 0x84 = Type B / C
byte 1      dataCount    number of fields
byte 2      blobCount    number of bitmap blobs
bytes 3-4   faceNumber   design id (u16, little-endian)

# fields, from byte 5
Type B/C    39 × 10 bytes   [type, blobIdx, x, y, w, h]   # x/y/w/h u16 LE
Type A      32 ×  6 bytes   [type, x, y, w, h, blobIdx]   # u8 fields

Our CLI reads this directly — moyoung inspect face.bin prints each field's type, name, rectangle and blob index. Verified against a dawft-built face: the 17 declared fields round-tripped exactly.

Flashing

Getting a face onto the watch

All of it rides GATT service 0xFEEA: 0xFEE2 for control, 0xFEE6 for image bytes, 0xFEE3 for the device's acks. Proven end-to-end on a MOY-ERJ3 (a 63 KB face, 259 chunks, completion ack received).

Verify identity.

Read the manufacturer characteristic 0x2A29; it must say MOYOUNG-V2. Subscribe to notifications on 0xFEE3.

Announce the size.

Write feea 20 09 74 + the image length (u32, big-endian) to 0xFEE2, then settle briefly.

Stream the image.

Write the bytes to 0xFEE6 in MTU-sized chunks. On Linux/BlueZ the MTU is often left at 23, so chunk to ~244 bytes — the watch reassembles by the announced total, so chunk size is free.

Finish & activate.

Wait for the notify feea 20 09 74 ff (transfer complete), send the finish + set-transfer commands, then feea 20 06 19 + slot to switch the watch to the new face.

# our standalone CLI (bleak, no phone / Da Fit app)
moyoung scan                          # find MoYoung (0xFEEA) watches
moyoung list-fields                   # print all 67 embeddable fields
moyoung inspect  face.bin             # parse a face .bin's fields
moyoung --address D8:69:.. upload-face face.bin   # flash it

What you can inject onto a face

Most face fields show the watch's own state (steps, HR, calories, battery) and can't be faked. Exactly these carry an arbitrary external value pushed over Bluetooth:

WEATHER_TEMP

A signed number (−128…127) via weather --temp. The cleanest slot — big and purpose-built. Drives any WEATHER_TEMP field.

slot 1 · clean

STEPS_GOAL

An integer (0…100k) via set-goal. A genuine second slot — drives the STEPS_GOAL field. (Overwrites the real step goal.)

slot 2 · clean

clock / date

Via set-time — hijackable like the GTX2 day-field trick, but it ticks and reformats, so it's messy. A last resort.

time-natured

not injectable

Steps, heart rate, calories, distance and watch battery are sensors the watch measures. Music/notification text has no face field.

Home Assistant

Driven from HA, over a BLE proxy

A HAOS custom integration (moyoung) runs the protocol inside Home Assistant via bleak, routed through an ESPHome Bluetooth proxy with active connections — so the HA server needs no local adapter, and any room with a proxy can reach the watch.

Entities

Battery %, steps, Location (nearest-proxy room) and Signal (RSSI) — with per-proxy RSSI as attributes.

sensors

Services

moyoung.set_time · weather · notify · music · find · set_goal · upload_face — the CLI surfaced as HA actions for automations.

actions

Room tracking

Location follows whichever proxy hears the watch strongest, updating as it roams. A dashboard adds a room-over-time graph + a live per-proxy RSSI breakdown.

nearest-proxy

Add-by-MAC gotcha

A proxy-forwarded advert drops its service UUIDs, so 0xFEEA auto-discovery misses the watch even though HA can connect. The config flow falls back to manual MAC entry.

A solar dashboard face

Both injection slots plus the clock turn the watch into a glanceable energy readout — the MoYoung analog of the GTX2 live grid-kW driver. HA keeps the two numbers live; the face ships once, no re-flash.

Live, no re-flash

The face installs once; HA pushes the two values whenever they change — SoC follows the battery, the rate follows the time-of-use schedule. It's the pattern the GTX2 uses for grid kW, here with two independent channels instead of one.

Context

How this differs from the GTX2

Different chip, different reach

The GTX2 is an Actions ATS3085 running Zephyr; this watch is a Realtek RTL8762-class MoYoung-v2 device (Da Fit app). Both let you smuggle live external values onto a custom face (see Control & inject above) — the GTX2 via clock fields, MoYoung via WEATHER_TEMP + STEPS_GOAL. The big difference: unlike the GTX2, custom firmware isn't reachable on MoYoung — its OTA path is Realtek's closed, signed channel. Custom faces and live injection, though, work great — all verified on-glass.
